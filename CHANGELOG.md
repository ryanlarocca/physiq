- **June 29, 2026 (pt 2)** — **In-app "Install Physiq" helper (Add to Home Screen).** Ryan wanted users to install the PWA without him explaining the iOS steps. The app was already a valid PWA (manifest `display:standalone` + apple meta tags), but iOS Safari has **no install API** (no `beforeinstallprompt`), so you can't trigger it from a button — only guide the user. **Built** a dismissible bottom banner ([`index.html`](./index.html)) that adapts per platform: **iOS Safari** → "Add" button opens an instructions sheet (Share-icon SVG → "Add to Home Screen"), with a note to open in Safari if in an in-app webview; **Android/Chromium** → captures `beforeinstallprompt` for a real **one-tap Install**; **already-installed** (`display-mode:standalone` / `navigator.standalone`) or previously-dismissed → never shows. SW `v6→v7`. **Tests:** `scripts/test-install.mjs` 9/9 (puppeteer, faithful per-platform sim — suppresses Chromium's native event so the iOS path tests like real Safari); UI smoke still 12/12. **Deployed:** GitHub Pages from `main`.

- **June 29, 2026** — **Passwordless OTP login + true per-user data + AI moved to Anthropic directly (multi-user launch).** Ryan wanted new users to sign up via the shared link and get their own isolated data; testers had reported "got the email but couldn't log in." **Root causes found:** (1) signup used password + email **confirmation**, and the confirmation link pointed at a dead-end `confirm-email.html` that threw away the token — plus PKCE breaks across in-app webviews; the DB showed **3 external users stuck `confirmed:false`**. (2) `seedFromDataJson()` injected Ryan's bundled `data.json` into **every empty account**, so new users got Ryan's weigh-ins/macros — the real "not separate data" bug (caught when a headless test user got seeded with Ryan's 186.2 weights). (3) AI food logging was already broken in prod — the `OPENROUTER_API_KEY` was dead ("User not found" from OpenRouter). **Fixes:** replaced password UI with **email OTP** (`signInWithOtp`→`verifyOtp`, 6-digit code typed back in the same tab — no link, works in any browser); one flow for signup+login; stuck users auto-recover by requesting a code. Configured Supabase OTP email template (`{{ .Token }}`, 6-digit) via the Management API. **Removed** `seedFromDataJson` (new users start empty) and the cross-user `migrateUnownedData` landmine. Switched both `parse-meal` (text) and `parse-meal-photo` (vision) from OpenRouter → **Anthropic Messages API** (`claude-sonnet-4-6`), set `ANTHROPIC_API_KEY` in Vercel. **JWT-gated** all 3 `physiq-api` endpoints (`verifyUser` against `/auth/v1/user`) so only signed-in users spend AI; client now sends the bearer token. Deleted the dead in-app Gemini-key field + `confirm-email.html`; bug reports now carry the real logged-in email; SW `v5→v6`. **Tests (all green):** RLS isolation 10/10 (reads/writes/spoof-blocked/anon-blocked, real user JWTs), proxy gating 4/4, headless UI 12/12 (OTP screen, app boot, **new-user-starts-empty**), live text+photo parse 200. Full Supabase CSV backup taken pre-ship (`backups/`, gitignored); test pollution cleaned, Ryan's data verified intact (265 weight / 723 macro). **Files:** [`index.html`](./index.html), [`sw.js`](./sw.js), `physiq-api/api/{parse-meal,parse-meal-photo,submit-bug,_auth}.js`, new `scripts/`. **Deployed:** proxy via Vercel (Anthropic + gated); app via GitHub Pages from `main`.

- **June 15, 2026** — **Weight chart Y-axis now zooms to actual weights — the goal line was stretching the chart.** Ryan flagged that the weight chart looked too "flat" — current weights cluster ~191–196 but the chart bottom was anchored down at 171 because the **goal weight of 175** was being treated as a data point, pulling `yMin` down and burning ~20 lbs of vertical space on a region with no data. **Root cause:** Chart.js's default auto-scaling considers every dataset (weights, 7-day avg, 30-day avg, and the horizontal goal line) when computing `min`/`max`, then adds its own padding — so a goal point well outside the actual weight cluster anchors the axis to itself. **Fix:** compute `yMin` / `yMax` manually from **visible weights + the 7-day moving average only** (goal explicitly excluded from the bounds calculation), apply a 15% buffer, `Math.floor` / `Math.ceil` to clean integers, and wire those into the y-axis scale config. The goal line still renders on the chart; it just no longer drags the axis around. **Files:** [`index.html`](./index.html) — bounds computation at ~lines 2840–2848, y-axis scale config a few lines below. **Commits:** `dd9cdf7` "Weight chart: dynamic Y-axis scaling to actual data range" + follow-up `0d34c6b` "fix(chart): exclude goal from Y-axis bounds so chart zooms to actual weights" (first pass still included goal in bounds; second pass fixed it). **Verified:** chart on live site now zooms tight to the 191–196 cluster with a small buffer; goal line at 175 still visible; works across 7D / 14D / 30D / All ranges. **Deployed:** GitHub Pages auto-deploy from `main`.
